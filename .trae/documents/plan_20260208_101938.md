我理解您的要求。遵循最佳实践，将业务逻辑封装在独立的 Hono 路由中，而不是依赖通用的 `/data` 接口。

### 1. 数据库 Schema 变更
在 `apps/web/prisma/schema.prisma` 中添加核心表结构：
- **game_rooms**: 包含房间状态、配置、房主信息。
- **game_players**: 管理玩家与房间的绑定关系及状态。

### 2. 后端 API 开发 (Hono)
创建独立的路由文件 `apps/web/src/server/api/routes/game.ts`，实现以下业务端点：
- **POST /rooms**: 创建房间（验证参数、初始化状态、自动将房主加入玩家列表）。
- **POST /rooms/:id/join**: 加入房间（校验房间人数、状态、防止重复加入）。
- **POST /rooms/:id/leave**: 离开房间（处理房主转移或房间销毁逻辑）。
- **POST /rooms/:id/start**: 开始游戏（校验权限和人数）。

并在 `apps/web/src/server/api/app.ts` 中注册 `/game` 路由。

### 3. 前端大厅重构 (Lobby)
使用 Ant Design 重写 `apps/web/src/app/lobby/page.tsx`：
- **数据获取**：列表展示仍可利用 Refine 的 `useList` (配合 RLS) 以获得最佳的查询性能和实时订阅能力。
- **业务操作**：**创建**和**加入**操作将不再直接写库，而是通过 Refine 的 `useCustomMutation` 调用上述自定义的 Hono API，确保业务规则在后端得到执行。
- **交互设计**：实现创建房间的 Modal 表单和房间列表的 Table 展示。

此方案既保证了数据操作的安全性（后端闭环），又保留了 Refine 前端开发的便利性。
